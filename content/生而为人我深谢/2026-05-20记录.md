## 原始回填
### User
Playwright做自动化的原理是什么？

### Assistant
Playwright 的核心原理可以理解成： 它不是“模拟鼠标键盘乱点”，而是“通过浏览器自动化协议，直接指挥浏览器内部对象”。 简单拆开说： 1. 它先启动一个真实浏览器 - 可以是 Chromium、Firefox、WebKit - headless 或有界面都可以 - 浏览器仍然是真浏览器，不是纯解析器 2. 它和浏览器建立控制通道 - 对 Chromium，主要走 CDP（Chrome DevTools Protocol） - 对 Firefox/WebKit，走各自的自动化协议 - 本质是一个进程间通信通道，通常是 WebSocket/pipe 一类 3. 它控制的是“页面对象” Playwright 直接操作： - Browser - BrowserContext - Page - Frame - Element/Locator 所以你写的： - click() - fill() - press() - goto() - waitForResponse() 其实是在调用浏览器内部能力，不是纯前台 UI 自动点按。 4. 它会自动等待 这是 Playwright 很强的地方。 比如你点一个按钮时，它不会只做“单击动作”： - 先等元素可见 - 等可点击 - 等不被遮挡 - 等页面稳定一点 - 之后再执行 这就是为什么它通常比“自己 sleep+点击”的脚本稳定。 5. 它会注入脚本帮助读取页面状态 Playwright 可以在页面上下文里执行 JS： - 读 DOM - 读 localStorage / sessionStorage - 读 cookie - 监听网络请求 - 拦截响应 - 处理 iframe、shadow DOM 6. 它还能控制网络层 比如： - 拦截请求 - 改 header - mock 接口 - 阻止图片/CSS 加载 - 记录 API 返回 所以 Playwright 不是单纯“UI 自动化工具”，而是“浏览器级自动化平台”。 一句话总结： Playwright = 启动真实浏览器 + 通过自动化协议控制浏览器内部状态 + 自动等待 + DOM/网络/页面级操作。 如果你愿意，我还可以继续给你讲两件更实用的： - Playwright 和 Selenium 的原理差别 - 为什么高对抗网站会识别 Playwright/headless
